| 组件 | 可行性 | 关键发现 |

## 一、ResNet 做视觉感知

### 结论：成熟方案，直接可用

**推荐模型：ResNet-50**（不是 ResNet-18 或更深的变体）

理由：
- ResNet-50 是植物病害识别领域使用最广泛的架构，2025 年绝大多数论文以此为基准
- 在 PlantVillage 数据集（38 类，54000+ 图像）上准确率 **98-99.5%**
- CPU 推理延迟 20-100ms，ONNX Runtime 优化后 15-40ms，完全够用
- Hugging Face 上有 `microsoft/resnet-50` 预训练模型，可直接微调

**关键限制：实验室 vs 真实田间**

| 场景 | 预期准确率 | 说明 |
|------|-----------|------|
| 实验室标准图片（PlantVillage） | 98-99.5% | 背景纯净、光照均匀 |
| 温室拍照（较可控） | 92-97% | 有干扰但相对可控 |
| 田间拍照（最真实） | 85-95% | 光照不均、背景杂乱、角度随意 |

**你的 MVP 建议：**
- 先用 ResNet-50 + PlantVillage 微调，验证视觉 MCP Server 的完整链路
- 后期用中国数据集（AI Challenger 2018 农作物病害数据集，约 5 万张，61 类）增强中文场景覆盖
- 部署时导出为 ONNX 格式，用 ONNX Runtime 做推理，CPU 上 15-40ms 完成一张图的分类

**可用的中国农业图像数据集：**
- AI Challenger 农作物病害数据集（2018）：~5 万张，61 类，中文标注
- 百度 AI Studio：搜"作物病害"可找到多个开放数据集
- 天池（阿里）：历史竞赛数据集
- 国家农业科学数据中心（data.agridata.cn）：需申请

## 二、Qwen 小模型处理传感器数据

### 结论：非常合适，是这个架构中最容易落地的一环

**推荐模型：Qwen2.5-3B-Instruct**

| 模型 | 参数量 | 量化后大小 | CPU 速度 | 最低内存 |
|------|--------|-----------|---------|---------|
| Qwen2.5-0.5B | 0.5B | ~400 MB | 极快 | 2 GB |
| Qwen2.5-1.5B | 1.5B | ~1.0 GB | 10-20 tok/s | 4 GB |
| **Qwen2.5-3B** | **3B** | **~2.0 GB** | **5-10 tok/s** | **8 GB** |
| Qwen2.5-7B | 7B | ~4.5 GB | 2-5 tok/s | 16 GB |

**为什么选 Qwen 而不是 Phi-3 或 Gemma：**
- Apache 2.0 开源，可商用、可修改、可私有化部署
- 中文能力远超 Phi-3 和 Gemma（支持 29+ 语言）
- JSON 结构化输出能力强，天然适合传感器数据 → 结构化 JSON 的任务
- 支持函数调用（function calling），可以作为 Agent 的子任务执行者
- Hugging Face 上有完整的模型权重和量化版本

**具体用法（传感器数据处理）：**

```
输入：温室A传感器原始读数
  温度33.1°C 湿度55% 土壤含水42% CO2 850ppm 叶面有水
  
输出（结构化JSON）：
{
  "alerts": ["温度偏高", "叶面结露"],
  "risk_level": "medium",
  "likely_disease_risk": "叶霉病（高湿+结露+温度适中）",
  "recommended_action": "立即通风降湿"
}
```

**部署方式：**
- 通过 Ollama 或 llama.cpp 本地运行，纯 CPU，无需 GPU
- 量化格式：GGUF Q4_K_M（~2GB）
- 你的 Windows 11 台式机如果内存 ≥16GB 就能流畅运行

**微调建议：**
- 用 Unsloth + QLoRA 微调，数据格式：传感器读数 → 专家诊断 JSON
- 微调数据不需要很多，500-1000 条高质量配对数据就能看到明显效果
- 框架：Unsloth（比传统 LoRA 快 2-5 倍）

## 三、GLM-5.2 做中央调控——关键修正

### GLM-5.2 不存在

调研确认：智谱 AI 发布的最新旗舰模型叫 **GLM-5**（2025年7月11日发布），有 GLM-5、GLM-5-Pro、GLM-5-Air、GLM-5-Flash 四个变体，但没有 "GLM-5.2" 这个版本。你之前提到的 "GLM-5.2" 可能是与其他信息源混淆了。

### GLM-5 的真实情况

| 属性 | 状态 |
|------|------|
| 架构 | MoE（混合专家），总参数量未公开（千亿级） |
| 开源 | **旗舰版不开源**，仅 API 调用（open.bigmodel.cn） |
| 可自部署 | **否**，只能通过智谱 API 使用 |
| 工具调用 | 原生支持，并行调用 + 多步编排 |
| Agent 能力 | 核心卖点，配合 AutoGLM 2.0 框架 |

### 替代方案

由于 GLM-5 旗舰版不开源、不可自部署，你需要替代方案：

| 方案 | 模型 | 开源 | 可自部署 | 工具调用 | 推荐度 |
|------|------|------|---------|---------|--------|
| A | **GLM-Z1-32B**（智谱开源推理模型） | 是 | 是（需 ~64GB 显存或量化） | 支持 | ★★★★ |
| B | **DeepSeek-V3 / R1** | 是 | 是 | 强 | ★★★★★ |
| C | **Qwen2.5-72B** 或 Qwen3 | 是 | 是 | 强 | ★★★★ |
| D | **GLM-5 API 调用**（不自部署） | 否 | 否 | 原生最强 | ★★★ |

**我的建议：**

如果你的目标是"数据主权 + 永不宕机"（你之前提过这个原则），那 GLM-5 API 调用不符合要求。

**首选方案 B：DeepSeek-V3**，理由：
- 完全开源（MIT 协议），可本地部署
- 推理能力强，适合复杂的多步农业决策
- 工具调用能力成熟
- 中文能力一流

**备选方案 A：GLM-Z1-32B**，理由：
- 智谱生态，与 GLM-5 同源
- 专注推理，适合"中央调控"角色
- 32B 参数量在量化后可以消费级硬件运行（4x RTX 4090 或 2x A100）

**折中方案 D：GLM-5 API + 本地降级**：
- 正常情况走 GLM-5 API（最强 Agent 能力）
- 网络断开时降级到本地 GLM-Z1-32B 或 DeepSeek
- 这其实呼应了你之前说的"Agent 的认识论"——知道什么时候自己能力不够，优雅降级

## 四、农作物栽培向量知识库——**应该建，而且是这个系统的灵魂**

### 为什么要建

你之前说过一个核心洞察：**"如果信息源全错，AI 无法得出正确答案。"**

向量知识库就是解决这个问题的：
- 它是系统的"可信记忆"——所有知识经过筛选和结构化，不是互联网上的二手信息
- 它让小模型也能做专业判断——Qwen 3B 自身不懂农业，但通过 RAG 检索知识库后，可以给出专家级建议
- 它是系统随时间进化的基础——新的病害案例、新的种植经验都可以持续入库

### 技术选型

**向量数据库：**
- MVP 阶段用 **Chroma**（纯 Python，零配置，5 分钟上手）
- 生产阶段迁移 **Milvus** 或 **Qdrant**（支持分布式、元数据过滤）

**Embedding 模型：**
- 首选 **BGE-large-zh-v1.5**（BAAI 智源研究院）——中文检索能力排名第一
- 长文档场景用 **BGE-M3**（支持 8192 token，适合完整的栽培指南）

**知识来源：**
- OpenKG.cn（中文开放知识图谱社区，有农业领域数据集）
- GitHub 上搜"农业知识图谱"——有多个现成的 Neo4j 项目
- AGROVOC（FAO 国际农业叙词表，支持中文，CC 协议）
- 国家农业科学数据中心（data.agridata.cn）

### 知识库结构设计

```
农作物栽培知识库
├── 作物病害库
│   └── {disease_name, crop, pathogen, symptoms[], conditions, treatment[], prevention[]}
├── 栽培技术库
│   └── {crop, growth_stage, technique, parameters, region, season}
├── 土壤管理库
│   └── {soil_type, pH, nutrients, suitable_crops[], amendments[]}
└── 虫害防治库
    └── {pest_name, target_crops[], damage, lifecycle, control_methods[]}
```

每个文档切片 300-800 token，带元数据标签（作物、地域、季节、类别），支持向量语义检索 + 结构化过滤的混合查询。

## 五、整合架构：四模型如何协作

```
                    用户/传感器输入
                         │
                         ▼
              ┌─────────────────────┐
              │   中央调控层          │
              │   DeepSeek-V3 /      │
              │   GLM-Z1-32B         │
              │   (推理+规划+编排)     │
              └──┬──────┬──────┬────┘
                 │      │      │
        ┌────────┘      │      └────────┐
        ▼               ▼               ▼
  ┌──────────┐   ┌──────────┐   ┌──────────┐
  │ 视觉感知  │   │ 传感器    │   │ 知识检索  │
  │ ResNet-50│   │ 处理      │   │ RAG      │
  │ (病害识别)│   │ Qwen 3B  │   │ BGE+Chroma│
  └──────────┘   └──────────┘   └──────────┘
        │               │               │
        └───────────────┼───────────────┘
                        ▼
              ┌─────────────────────┐
              │  结构化诊断报告       │
              │  + 置信度            │
              │  + 环境预警          │
              │  + 处置建议          │
              └─────────────────────┘
```

### 各模型的角色分工

| 模型 | 角色 | 对应你之前的比喻 | 频率 | 成本 |
|------|------|-----------------|------|------|
| ResNet-50 | 视觉感知 | "眼睛" | 每次有新图片时 | 极低（CPU 15-40ms） |
| Qwen2.5-3B | 传感器数据结构化 + 异常检测 | "神经反射弧" | 每次收到传感器数据 | 低（CPU 5-10 tok/s） |
| DeepSeek/GLM-Z1 | 中央推理、多步规划、工具编排 | "大脑" | 每次用户查询 | 中（需要 GPU 或 API） |
| BGE + Chroma | 知识检索 | "长期记忆" | 每次推理时被调用 | 极低（毫秒级） |

### 与你之前 MVP 的衔接

你已经构建的 Agent Router + MCP Server 架构完全兼容这个多模型方案：

- `sensor_mcp_server.py` 中加入 Qwen 3B 做数据结构化和异常预判
- 新建 `vision_mcp_server.py`，内部用 ResNet-50 做图像分类
- `knowledge_mcp_server.py` 中把硬编码 JSON 替换为 Chroma 向量检索
- Agent Router 的意图识别和编排逻辑替换为 DeepSeek/GLM-Z1 的工具调用

**不需要重写架构，只需要替换组件。**
