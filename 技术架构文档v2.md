## 五大模块

### 1. ResNet — 视觉感知

**职责：** 对温室摄像头/手机拍摄的作物图片进行病害识别

| 项目 | 选型 |
|------|------|
| 模型 | ResNet-50（ImageNet 预训练 + PlantVillage 微调） |
| 准确率 | 实验室 98-99.5%，温室实际 92-97% |
| 推理速度 | CPU 15-40ms（ONNX Runtime），GPU 1-5ms |
| 输出 | 病害类别 + 置信度 |
| 数据集 | PlantVillage（国际）+ AI Challenger 2018（中文，5万张，61类） |
| 部署格式 | ONNX（跨平台，CPU 友好） |

**MCP 接口：**
- `diagnose_image(image_path)` → `{disease, confidence, top3_matches[]}`

---

### 2. LSTM — 传感器异常检测

**职责：** 对传感器时序数据做实时异常检测，发现异常模式后触发预警

| 项目 | 选型 |
|------|------|
| 模型 | 双层 LSTM + Dropout |
| 输入 | 滑动窗口的多变量时序（温度、湿度、土壤、CO2、光照） |
| 输出 | 异常分数 0-1，超过阈值触发告警 |
| 训练数据 | 正常运行期间的传感器历史数据 |
| 推理速度 | <5ms（纯 CPU，PyTorch） |

**检测的异常模式：**
- 温度骤升/骤降（可能设备故障或通风异常）
- 湿度持续偏高（病害风险预警）
- 传感器信号漂移或中断
- 多传感器交叉矛盾（如温度升但湿度也升——不符合物理规律）

**MCP 接口：**
- `check_anomaly(greenhouse_id)` → `{anomaly_score, abnormal_sensors[], pattern_description}`
- `get_anomaly_history(greenhouse_id, hours)` → 时序异常记录

---

### 3. Qwen — 环境数据解释

**职责：** 将传感器原始数据 + LSTM 异常检测结果转化为人类可读的语义分析

| 项目 | 选型 |
|------|------|
| 模型 | Qwen2.5-3B-Instruct（GGUF Q4_K_M 量化，~2GB） |
| 运行环境 | 纯 CPU，Ollama 或 llama.cpp |
| 输入 | 结构化传感器 JSON + 异常标记 |
| 输出 | 自然语言分析报告 + 建议 |

**为什么不用 Qwen 做异常检测：**
LSTM 擅长"这个数值序列的模式不对"（时序模式识别），Qwen 擅长"这意味着什么、该怎么办"（语义理解）。让 LSTM 做前端过滤，只把可疑数据交给 Qwen 解释，既省算力又更准。

**MCP 接口：**
- `interpret_conditions(sensor_data, anomaly_info)` → `{analysis, risk_level, explanation}`

---

### 4. RAG — 农业知识库

**职责：** 为所有决策提供可信的农业知识支撑

| 项目 | 选型 |
|------|------|
| 图数据库 | Neo4j（基于 qq547276542/Agriculture_KnowledgeGraph 项目） |
| 向量检索 | Chroma（MVP）→ Milvus/Qdrant（生产） |
| Embedding | BGE-large-zh-v1.5（中文检索能力最强） |
| 知识来源 | 农业知识图谱 + 栽培指南 + 病害数据库 + 植保手册 |

**知识库结构：**
```
├── 作物病害（病名、病原、症状、发病条件、防治方法）
├── 虫害防治（虫名、危害特征、生活史、防治方法）
├── 栽培技术（整地、播种、水肥管理、收获）
└── 土壤管理（土壤类型、pH、养分、改良建议）
```

**查询方式：**
- 语义检索：用户问"叶片发黄怎么办"→ 向量相似度匹配
- 图谱查询：A→B→C 关系链（番茄 → 叶霉病 → 多菌灵 → 500倍液）
- 混合查询：先向量召回候选，再用图谱关系验证和补充

**MCP 接口：**
- `search_knowledge(query, crop, domain)` → 结构化知识片段
- `get_disease_chain(disease_id)` → 完整的关系链（病原→症状→条件→防治）

---

### 5. DeepSeek — 任务规划和决策

**职责：** 理解用户意图，编排其他四个模块的调用顺序，综合所有信息做出最终决策

| 项目 | 选型 |
|------|------|
| 模型 | DeepSeek-V4-Pro（API 调用） |
| API | `https://api.deepseek.com`，OpenAI 兼容格式 |
| 核心能力 | 原生工具调用（function calling），并行调用，多步编排 |
| 备选 | DeepSeek-V4-Flash（更快更便宜，复杂任务稍弱） |

**决策流程示例（用户说"番茄叶子不对劲"）：**

```
DeepSeek 收到查询
  │
  ├─ 调用 LSTM: check_anomaly("gh-a") → 发现湿度持续偏高
  ├─ 调用 ResNet: diagnose_image(photo) → 叶霉病 92%
  ├─ 调用 RAG: search_knowledge("叶霉病", "tomato") → 防治方案
  ├─ 调用 Qwen: interpret_conditions(sensor_data, anomaly) → 环境分析
  │
  └─ 综合所有结果 → 输出诊断报告 + 处置建议 + 置信度
```

## MCP Server 清单

| MCP Server | 职责 | 核心工具 | 底层模型 |
|------------|------|---------|---------|
| `vision-mcp` | 视觉感知 | `diagnose_image` | ResNet-50 |
| `sensor-mcp` | 传感器数据 + 异常检测 | `get_reading`, `check_anomaly` | LSTM |
| `interpreter-mcp` | 环境数据语义解释 | `interpret_conditions` | Qwen 3B |
| `knowledge-mcp` | 知识检索 | `search_knowledge`, `get_disease_chain` | BGE + Neo4j |
| `weather-mcp` | 气象服务 | `get_forecast`, `get_alerts` | API/模拟 |
